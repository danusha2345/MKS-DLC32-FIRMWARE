# Удалённое управление: токен станка + адрес сервера + имя станка

Фича для работы парка станков через центральный сервер. Вместо того чтобы сервер
искал платы в локальной сети, **плата сама устанавливает исходящее TCP-соединение
к серверу** и первой строкой отправляет авторизационные данные (токен станка,
человекочитаемое имя, MAC). Дальше канал работает как обычная telnet-сессия GRBL:
сервер шлёт любые команды (G-код, `$`-команды, realtime `?` / `!` / `~`),
плата отвечает в этот же сокет.

## Параметры (хранятся в NVS, переживают перепрошивку и перезагрузку)

| Параметр | Команда GRBL (terminal/telnet/USB) | Команда ESP (WebUI/HTTP) | Формат |
|---|---|---|---|
| Адрес сервера | `$Remote/Server=192.168.1.50:9000` | `[ESP150]192.168.1.50:9000` | `host:port`, хост — IP или DNS-имя |
| Токен станка | `$Remote/Token=eyJhbGciOi...` | `[ESP151]eyJhbGciOi...` | произвольная строка до 250 символов (JWT помещается) |
| Имя станка | `$Remote/Name=Frezer ceh 2` | `[ESP152]Frezer ceh 2` | до 64 символов, **только латиница/цифры/пробелы** (байты ≥0x80, т.е. кириллицу, ядро GRBL съедает как realtime-команды); не использовать `,` и `]` (сломают разбор hello) |

- Имя станка — чисто для человекочитаемости: оператор по нему понимает, что это
  за станок. Плата его не интерпретирует, просто предъявляет серверу в hello.
- Посмотреть текущее значение: `$Remote/Server` / `$Remote/Token` / `$Remote/Name`
  без `=` (или `[ESP150]` / `[ESP151]` / `[ESP152]`).
- Выключить функцию: задать пустой адрес — `$Remote/Server=`.
- Команды можно отправлять по любому каналу: USB-serial, telnet (порт 23), WebUI-терминал,
  HTTP (`http://<ip>/command?plain=[ESP150]host:port`).
- Изменения подхватываются на лету: адрес читается перед каждой попыткой подключения,
  перезагрузка не обязательна (если соединение уже установлено со старым адресом —
  разорвать его со стороны сервера или перезагрузить плату).
- ⚠️ Лимит длины строки команды GRBL — 256 символов, поэтому через `$Remote/Token=...`
  влезает токен до ~240 символов. Реальный предел значения — 250.

## Протокол подключения

1. Плата подключена к Wi-Fi (режим STA). Если задан `Remote/Server`, она пытается
   открыть TCP-соединение к серверу (таймаут 2 с, повтор каждые 15 с — и после обрывов тоже).
2. Сразу после установки соединения плата отправляет одну строку:

   ```
   [HELLO:token=<значение Remote/Token>,name=<значение Remote/Name>,mac=<MAC платы>]\r\n
   ```

   По `token` сервер аутентифицирует/детектит станок, `name` — человекочитаемое
   имя для отображения оператору, `mac` — уникальный идентификатор железа
   (полезно при дублировании токена).
3. Затем (если включён welcome) придёт стандартный баннер GRBL (`Grbl 1.x ...`).
4. Дальше — полный двусторонний обмен как в telnet: сервер пишет команды строками,
   плата отвечает `ok`/`error:N`/отчётами. Статус запрашивается символом `?`
   (без перевода строки), ответ — `<Idle|MPos:...>`.

В прошивке соединение занимает один из `TELNET_CLIENTS_TOTAL` (3) клиентских слотов,
поэтому работает весь существующий конвейер ввода/вывода GRBL без изменений.
Локальный входящий telnet и WebUI продолжают работать параллельно.

## Сообщения платы (видны во всех подключённых консолях)

- `[MSG:REMOTE Connected to host:port]` — соединение установлено, hello отправлен;
- `[MSG:REMOTE Disconnected]` — связь потеряна, начались повторные попытки;
- `[MSG:REMOTE Bad server address, expected host:port]` — адрес не парсится (нет `:порт`).

Неудачные попытки подключения (сервер недоступен) не логируются, чтобы не засорять консоль.

## Минимальный сервер для проверки (netcat)

```bash
nc -l -p 9000        # ждём; плата сама подключится в течение ~15 с
# увидим: [HELLO:token=...,mac=...]
# вводим: ?            → ответ <Idle|...>
# вводим: $G           → модальные группы, ok
```

## Загрузка G-code файлов на SD-карту

Файлы заливаются по HTTP (тот же эндпоинт, что у WebUI и Android-приложения):

```bash
curl "http://<ip-платы>/upload?path=/" -F "path=/" \
     -F "myfile=@job.gcode;filename=/job.gcode"
# запуск:  [ESP220]/job.gcode  (через сокет или /command?commandText=...)
```

Важные ограничения (проверено на плате):

- **Во время выполнения задания с SD загрузка блокируется намеренно** — карта
  занята чтением исполняемого G-code, запись оборвала бы рез. Заливать файлы
  до запуска задания; после завершения загрузка снова работает. Сейчас плата
  при этом обрывает соединение без ответа (а не отдаёт `Busy`) — учитывать
  в клиенте.
- Карта — **только FAT32** (exFAT не монтируется; карты >32 ГБ из магазина
  обычно exFAT — переформатировать).

## Замечания по безопасности

Канал — голый TCP без TLS (как и стоковый telnet): токен передаётся открытым текстом.
Для работы через интернет закрывать трафик на уровне сети (VPN/WireGuard до сервера)
или держать сервер в той же локалке. JWT можно валидировать на сервере по подписи —
плата его не интерпретирует, просто хранит и предъявляет.

## Реализация (для разработчиков)

- `src/WebUI/RemoteClient.{h,cpp}` — модуль исходящего подключения (новый);
- `src/WebUI/WebSettings.{h,cpp}` — настройки `Remote/Server` (ESP150), `Remote/Token` (ESP151),
  `Remote/Name` (ESP152);
- `src/WebUI/TelnetServer.{h,cpp}` — `attach_client()` сажает исходящий сокет в свободный
  telnet-слот; слоты теперь инициализируются даже при `$Telnet/Enable=0`;
- `src/WebUI/WifiServices.cpp` — begin/handle/end модуля в жизненном цикле Wi-Fi-сервисов.
